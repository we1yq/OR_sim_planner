# 3-GPU Online Replay Plot Specification

This note defines the plots and plotting rules for the 3-GPU short online
experiment. The experiment follows the same visual and semantic rules used for
the 24-hour online replay, but uses the 3GPU trace generated by
`eval/3gpu_test/online_3gpu_trace.py`.

## Experiment Input

- Trace: `eval/3gpu_test/results/online_3gpu/<timestamp>/request_rate_30min.csv`
- Trace generator: `eval/3gpu_test/online_3gpu_trace.py`
- Replay/plot program: `eval/3gpu_test/online_3gpu_replay.py`
- Duration: 3.5 hours of demand transitions, plus a final zero-demand target
  point used to return the system to the empty/base state.
- Epoch length: 30 minutes.
- Epoch count: 8 demand points.
- Transition count: 7 transitions.
- GPU bound: every target demand point must fit within 3 GPUs.
- Validation files:
  - `slicewise_target_gpu_probe.csv`
  - `parvagpu_target_gpu_probe.csv`
  - `trace_metadata.json`

The generated trace currently uses source epochs `[1, 5, 13, 15, 37, 45, 47]`
from the 24h trace, followed by one all-zero shutdown row. The trace-selected
points preserve visible demand and workload-mix shifts while keeping target
allocations within 3 GPUs; the shutdown row drives all demand to zero after the
experiment ends.

By default, `online_3gpu_trace.py` reads the verified source trace
`eval/results/online_24h/20260710-021814`. This avoids accidental changes when
`eval/results/online_24h/latest_path.txt` is updated by other runs. To build the
3GPU trace from another 24h source, set:

```text
ONLINE_3GPU_SOURCE_TRACE_DIR=/path/to/eval/results/online_24h/<timestamp>
```

## Required Figures

### 1. Request Rate Trace

Output names:

- `eval/3gpu_test/figures/online_3gpu_request_rate.{pdf,png}`
- `eval/3gpu_test/figures/online_3gpu_request_rate_normalized.{pdf,png}`

Purpose:

- Show the 3.5-hour workload demand trace used by the experiment, including the
  final shutdown-to-zero point.
- Plot all seven workloads as step functions.
- Use log scale for raw request rates because vision and LLM rates differ by
  orders of magnitude.
- The normalized version should plot `rate / mean(rate)` to show shape and mix
  changes without scale dominating.

Rules:

- X-axis can be real experiment time in hours for the demand-only trace.
- Legend must not cover the data curves.
- Keep workload colors consistent with the 24h figures.

### 2. Active GPU Count Over Time

Output name:

- `eval/3gpu_test/figures/online_3gpu_active_gpu_count.{pdf,png}`

Purpose:

- Show resource occupancy during the replay.
- Compare SliceWise and Jormungandr.
- Show whether a method needs temporary extra GPUs during transitions.

Rules:

- This is a resource-count figure, not a serving-capacity figure.
- During normal epoch time, plot the target GPU count for that epoch.
- During a transition window, plot `peak_active_gpu`.
- Do not show action-level drops caused by route activation/deactivation timing.
- A downward step is allowed only when the target allocation really shrinks.
- The y-axis label should be `Active GPU count`.

### 3. Extra GPU Box Plot

Output name:

- `eval/3gpu_test/figures/online_3gpu_extra_gpu.{pdf,png}`

Purpose:

- Quantify temporary transition headroom.

Definition:

```text
extra_gpu = peak_active_gpu_during_transition
            - max(current_target_gpu_count, next_target_gpu_count)
```

Rules:

- Plot SliceWise and Jormungandr.
- This is a box plot over the seven transitions, including the final shutdown
  transition.
- Negative values are clamped to zero.
- If all values are zero, still keep the figure because it directly supports the
  claim that the transition does not require extra GPUs.

### 4. Min Service / Demand Timeline

Output name:

- `eval/3gpu_test/figures/online_3gpu_service_rate_min_ratio.{pdf,png}`

Purpose:

- Show the minimum service-capacity coverage across workloads over time.

Definition:

```text
service_rate_ratio = active_routed_capacity / committed_demand
committed_demand   = min(source_demand_rate, target_demand_rate)
```

For steady-state intervals after the transition completes, use the target demand
as the committed demand.

Rules:

- Plot SliceWise and Jormungandr.
- Draw a horizontal reference line at `1.0`, labeled `Demand`.
- Values below `1.0` mean the transition violates the committed demand.
- For final SliceWise results, do not keep a figure produced by forcing an
  invalid transition plan to execute. If the capacity check fails, fix the
  transition plan or mark the transition as invalid rather than presenting the
  forced dip as a valid result.
- Cap the plotted y-value at a readable upper bound, e.g. `2.5`, as in the 24h
  figure.

### 5. Per-Workload Active Capacity

Output names:

- `eval/3gpu_test/figures/online_3gpu_service_rate_resnet50_image.{pdf,png}`
- `eval/3gpu_test/figures/online_3gpu_service_rate_vgg16_image.{pdf,png}`
- `eval/3gpu_test/figures/online_3gpu_service_rate_vit_base_image.{pdf,png}`
- `eval/3gpu_test/figures/online_3gpu_service_rate_gpt2_p64_o64.{pdf,png}`
- `eval/3gpu_test/figures/online_3gpu_service_rate_gpt2_p512_o512.{pdf,png}`
- `eval/3gpu_test/figures/online_3gpu_service_rate_llama_p1024_o128.{pdf,png}`
- `eval/3gpu_test/figures/online_3gpu_service_rate_llama_p2048_o64.{pdf,png}`

Purpose:

- Show, for each workload, whether active routed capacity covers demand through
  all transitions.

Rules:

- Plot SliceWise active routed capacity.
- Use legend label `Active capacity`, not `Service rate`.
- Plot committed demand as a dashed green line labeled `Demand rate`.
- The blue transition background must align with the same transition windows as
  every other time-axis figure.
- If capacity temporarily decreases but stays above demand, it is acceptable.
- If capacity drops below demand in a final SliceWise figure, treat it as a
  planner/simulator bug unless the plot is explicitly marked as a rejected
  baseline or debug result.

### 6. Transition Makespan Box Plot

Output name:

- `eval/3gpu_test/figures/online_3gpu_transition_makespan.{pdf,png}`

Purpose:

- Compare how long transitions take.

Rules:

- Plot SliceWise and Jormungandr.
- Use seconds as the y-axis unit.
- This is a box plot over the seven transitions, including the final shutdown
  transition.

### 7. SLO Violation Duration Box Plot

Output name:

- `eval/3gpu_test/figures/online_3gpu_slo_violation.{pdf,png}`

Purpose:

- Summarize capacity-shortfall time during transitions.

Rules:

- The metric is measured in seconds.
- A valid SliceWise transition should be zero if the capacity-preserving DAG is
  correct.
- Do not obtain this plot by disabling Stage3 capacity checks and forcing
  invalid action sequences to execute.
- If a baseline lacks a capacity-preserving transition order, report the
  violation honestly, but make clear that it is a transition-policy failure, not
  normal serving noise.

### 8. Transition Action Counts

Output name:

- `eval/3gpu_test/figures/online_3gpu_transition_actions.{pdf,png}`

Purpose:

- Show the composition of transition work.

Rules:

- Plot stacked bars for SliceWise and Jormungandr.
- Use the same action categories and colors as the 24h plot:
  - create pod
  - delete pod
  - create MIG
  - delete MIG
  - reconfigure
  - migrate
- Aggregate over the seven transitions unless a per-transition panel is needed.

### 9. Planner Makespan Box Plot

Output name:

- `eval/3gpu_test/figures/online_3gpu_planner_makespan.{pdf,png}`

Purpose:

- Show planning overhead for the short 3GPU replay.

Rules:

- Plot SliceWise and Jormungandr.
- Use seconds as the y-axis unit.
- This plot is secondary; include it if space allows or place it in appendix.

### 10. Non-Serving MIG Slices

Output name:

- `eval/3gpu_test/figures/online_3gpu_unused_slices.{pdf,png}`

Purpose:

- Show how much allocated GPU slice capacity is not serving workloads.

Definition:

```text
non_serving_slices = 7 * gpu_count - workload_serving_slices
```

Rules:

- The y-axis label must be `Non-serving MIG slices`.
- Do not call it only `Unused slices`, because the value may include both truly
  idle slices and A100 MIG geometry holes such as `unusable` slices.
- If possible, later split this into `idle slices` and `unusable slices`.

### 11. Allocation Similarity

Output name:

- `eval/3gpu_test/figures/online_3gpu_allocation_similarity.{pdf,png}`

Purpose:

- Show how much the target allocation changes between adjacent epochs.

Rules:

- Plot SliceWise, Jormungandr, and ParvaGPU-MIG if available.
- Use the same metrics as the 24h replay:
  - partition similarity
  - exact workload similarity
- This plot is optional for the main 3GPU story, but useful for explaining why
  transition action counts differ.

## Global Time-Axis Rules

Apply these rules to every replay figure with a time axis:

- Use compressed display time: each 30-minute epoch is shown as 300 seconds.
- The x-axis label should be only `Time (s)`.
- Explain the compression in the figure caption, not in the axis label.
- Transition intervals must use the same light-blue background:

```text
color = "#DDE7FF"
alpha = 0.70
label = "Transition"
```

- For figures comparing SliceWise and Jormungandr, the blue transition span for
  an epoch should cover the maximum transition makespan across the compared
  methods. This keeps the transition background semantically consistent.
- For method-specific figures, using the method-specific transition span is
  acceptable, but avoid mixing span definitions within the same figure.

For this 3GPU experiment:

```text
actual epoch length       = 1800 s
display epoch length      = 300 s
display total time        = 8 * 300 s = 2400 s
transition-bearing epochs = 1..7
final transition          = shutdown to zero demand
```

## Capacity and SLO Rules

- During a transition from source demand to target demand, the committed demand
  for each workload is:

```text
min(source_demand_rate, target_demand_rate)
```

- The active capacity is the sum of throughput from serving instances whose
  routes are active.
- A valid capacity-preserving transition must keep:

```text
active_capacity(workload) >= committed_demand(workload)
```

- If stopping/deleting an old instance would violate this rule, the action DAG
  must first schedule a producer action that restores capacity:

```text
place/create target instance
activate target route
then deactivate/drain/delete old instance
```

- If the target producer physically overlaps with the old instance, use a bridge
  or extra GPU only when necessary.
- Do not disable capacity checks to generate paper figures.
- A graph with active capacity dropping below demand is only acceptable if it is
  intentionally labeled as a rejected/debug/baseline-violation result.

## Naming Rules

- Use `online_3gpu_` as the prefix for all 3GPU figures and replay CSVs.
- Keep workload suffixes identical to the workload names:
  - `resnet50_image`
  - `vgg16_image`
  - `vit_base_image`
  - `gpt2_p64_o64`
  - `gpt2_p512_o512`
  - `llama_p1024_o128`
  - `llama_p2048_o64`
- Save both PDF and PNG for every figure.

## Main-Paper Recommendation

For the main paper, the 3GPU experiment should probably show:

1. request-rate trace,
2. active GPU count,
3. min service / demand,
4. extra GPU box plot,
5. transition makespan or action counts.

The per-workload service-rate plots, planner runtime, non-serving slices, and
allocation similarity can go to appendix unless one of them reveals the key
3GPU behavior.
